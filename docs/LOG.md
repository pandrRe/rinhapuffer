Walks 14 columns × 2048-row chunks, encodes via `q = clamp(@round((f - min) * scale), 0, 65535)` into a 4 KiB stack `[2048]u16` buffer, streams each chunk through `writeStreamingAll`. - Phase 6 — HTTP/1.1 server skeleton (blocking, hand-rolled, libc-direct)
  - 6.1: new module `src/http.zig`. Drops `std.Io.net` entirely — uses `std.c.{socket, bind, listen, accept, read, write, writev, close, setsockopt, getsockname, connect}` directly with `std.posix.{AF, SOCK, SOL, SO}` constants. Mirror of `fast_json.zig`'s `posix`/`libc` aliasing pattern. Two module-level static buffers in BSS: `read_buf: [8192]u8` (covers worst-case ~3.5 KB payload + ~512 B headers + headroom) and `head_buf: [256]u8` (response head tops out around 150 B). Single-threaded accept loop guarantees at most one in-flight connection, so sharing across requests is safe. **Zero per-request, per-connection, or per-accept allocation** — the only `malloc` in the entire serve loop is whatever libc does internally on `accept`.
  - 6.2: hand-rolled `parse(bytes) ParseError!Request`. Schema-rigid: `GET` and `POST` only (`error.UnsupportedMethod` otherwise), `HTTP/1.1` only (`error.UnsupportedVersion`), `Content-Length` mandatory on POST (`error.MissingContentLength`), case-insensitive `Content-Length` and `Connection` header names + case-insensitive `close` value match for connection close detection. All slices in the returned `Request` alias the input buffer — zero copy. Header overflow without `\r\n\r\n` → `HeadersOversize`; declared body length exceeding what's buffered → `BodyTooLarge`. 10 unit tests cover well-formed GET/POST, mixed-case `Connection: Close`, mixed-case header names, every error variant.
  - 6.3: response writer. `format_head(buf, resp) ![]u8` is the testable pure-byte function — uses `std.fmt.bufPrint` with one of four format strings depending on `(empty body, keep_alive)`. `Content-Type` only emitted when body is non-empty; `Connection: close` only when `keep_alive == false`. `write_response(fd, resp)` calls `format_head` then `writev(fd, [head_iov, body_iov])` for non-empty bodies (one syscall for status + headers + body) or plain `write(fd, head)` when body is empty. Both wrappers loop on `EINTR` and handle short writes by advancing the iov base/len. 4 unit tests assert exact wire bytes for 200/keep-alive/body, 200/keep-alive/empty, 400/close/empty, 200/close/body.
  - 6.4: per-connection `serve(client_fd, handler)` loop. `read_until_request` blocks on `libc.read` until `\r\n\r\n` lands, sniffs `Content-Length` directly off the buffer to know how much body to wait for, then reads until total bytes are buffered. `EINTR` retried; clean peer-close before any bytes → `.closed` (signals serve to return); peer-close mid-request → `.err`; total > `REQ_BUF_SIZE` → `.err` (caller writes 413+close). After each successful handler dispatch, leftover bytes from a pipelined chunk are slid to the front of `read_buf` (rare on local TCP). Loop exits on peer close, parse error, write error, or `keep_alive == false` from the response.
  - 6.5: integration test in `http.zig`. `bind_listen(0)` lets the kernel pick an ephemeral port; `local_port(fd)` recovers it via `getsockname`. `std.Thread.spawn` runs `serve_one` (test helper that accepts exactly one connection then returns). Test client opens a raw libc socket, sends `GET /ready` over keep-alive followed by `POST /fraud-score` with `Connection: close`, reads until peer close. Asserts the response stream contains both 200s, `Content-Length: 0` for `/ready`, `Content-Length: 35` + the stub JSON for `/fraud-score`. **No `std.http.Client`** — same drop-out-of-std stance as the server side.
  - 6.6: `main.zig` becomes the production accept loop. Calls `http.bind_listen(9999)` once, then loops `accept_one` → `serve` → `close`. `init.io` unused (Phase 7 will use it to load `dataset.bin` once at startup). `dispatch` is a static function: `GET /ready` → 200 empty, `POST /fraud-score` → 200 + stub JSON, anything else → 404. `keep_alive` is propagated from request to response so the server honors `Connection: close` end-to-end. `src/root.zig` re-exports `http`.

- Phase 7 follow-up — first end-to-end run against the rinha k6 harness + cheap-experiment ladder for failure-rate
  - k6 v1.7.1 against `rinha-de-backend-2026/test/test.js` (ramping arrival rate 1→900 RPS over 120s, `preAllocatedVUs: 100, maxVUs: 250`, 54,100 test entries with 44.47% fraud, 1.47% edge cases). Each VU re-uses its underlying connection across iterations (k6 default keep-alive).
  - **Baseline (Phase 7 as shipped, `LISTEN_BACKLOG = 128`, keep-alive on): final score 0** — `failure_rate = 16.21%` triggered the `>15%` detection cut (`SCORE_DET_CORTE = -3000`). 8,439 timeouts out of ~54k attempts. p99 = 0.74 ms on the requests that completed; classification accuracy 99.5% on the served subset (FP=115, FN=125, TP=19,977, TN=24,891).
  - Diagnosis: at 250 concurrent VUs vs a single-threaded blocking accept loop, `serve()` enters its keep-alive `read_until_request` and **blocks on read** for whichever VU is currently active, starving 249 other accepted connections from being served. Bigger backlog only helps the queue; head-of-line blocking is the actual cap.
  - Cheap experiment 1: bump `LISTEN_BACKLOG` 128 → 1024. **Score 0 → 1,774.79** (timeouts 8,439 → 7,520, failure 16.21% → 14.48%, just under the cut). p99 = 0.69 ms. The cap was real but secondary; the bigger bottleneck remained head-of-line blocking on idle keep-alive reads.
  - Cheap experiment 2: disable keep-alive at the response layer (close-after-each). **Score 1,774.79 → 4,122 ✅** (timeouts 7,520 → **0**, failure 14.48% → 0.55%). Cost: p99 0.69 ms → 0.99 ms from the extra TCP connect per request — peanuts inside the 1 ms budget. Confirmed the head-of-line theory: with the connection lifetime bounded by the response, the server drains the accept queue at ~15k served/sec while demand is only 900 RPS.
  - Landed: comptime build option `-Dkeep-alive=true|false` (default `false`) → `handler.KEEP_ALIVE: bool = build_options.keep_alive`. Branch is comptime so the hot path costs nothing in either build. Tests assert against the const so they pass under both flavors. Startup line prints `(keep-alive on/off)` so the operator can see which binary is loaded.
  - Diagnostic experiments for the residual 0.55% failure rate (FP=149, FN=150 → goal: drive these toward 0):
    - **Hypothesis A — query is not L2-normalized but dataset rows are; metric mismatch.** Tried: L2-normalize the query inline before `cosine_topk_q_ivf`. Result: **bit-identical** FP/FN counts. Re-derived: when dataset rows have `|r|=1` (parse-time invariant), `cosine(q, r) = q·r / |q|`; dividing by `|q|` is constant across rows, so it's order-preserving and the top-K *indices* are unchanged. Mathematically a no-op for ranking — should have been caught at the whiteboard. **Hypothesis A falsified.**
    - **Hypothesis B — IVF approximate recall on the real query distribution.** Phase 5's reported recall@5 = 1.000 was on dataset-row + jitter queries (each query trivially lands in its own cluster); k6 queries don't share that distribution. Tried: bump `PROBE_CLUSTERS` 8 → 64 (8× more clusters scanned per query, asymptotically approaching brute force). Result: **FP=149, FN=149 (–1 of 299)** — essentially noise. p99 also blew up to 10.4 ms (cold mmap pages over 184k rows scanned + the inline-unrolled `sift_in_n` with N=64 generating a much fatter code path), score dropped to 3,107. **Hypothesis B falsified — IVF at PROBE=8 is already producing the brute-force top-K answer for the k6 query distribution.**
  - Conclusion: the 0.55% errors are either (a) inherent k=5 voting noise near the decision boundary, or (b) the **dataset L2-normalization in `transform_reference.parse_into` itself diverges from the spec's reference Euclidean distance**. Row norms vary substantially (rows with `last_transaction:null` carry two −1 sentinels → `|r| ≈ 1.7–2.0`; rows with real priors → `|r| ≈ 1.0–1.3`), so the unit-norm squashing is non-uniform across rows and changes neighbor ranking vs raw-feature Euclidean. Resolving this requires a real refactor: drop normalization from `parse_into`, regenerate `dataset.bin` (~50s prep), update `cosine_topk_q_ivf` to compute true Euclidean (`dist² = ||q||² + ||r||² − 2 q·r`; `||q||²` is row-constant and droppable; `||r||²` computed on-the-fly inside the same SIMD loop that already reads the 14 features for `q·r`).
  - Diagnostic landed: `scripts/diagnose.py` (numpy + requests). For every test-data entry, POSTs to the live server (whatever ranking the binary does today) and compares to `expected_approved`. On every mismatch, also runs a brute-force raw-Euclidean prediction in numpy over the un-normalized 3M references (`np.einsum("ij,ij->i", diff, diff)` → `argpartition` top-5 → labels[top].sum). Mirrors `payload.zig::vectorize` inline so it doesn't depend on the server exposing the query vector. ~75 sec wall (12 sec parse references.json + 54k HTTPs + ~300 numpy brute-force scans). Picked Python over a Zig diagnostic because (a) numpy is one-line for the brute-force, (b) testing the live server avoids drift between diagnostic and production, (c) one-off code; doesn't belong in `src/`.
  - **Diagnostic verdict (decisive):** of our 299 mismatches, **raw-Euclidean was correct on 298 of them**; only 1 was an inherent edge-case both methods miss. The dataset L2-normalization is unambiguously the source of every fixable FP/FN. Per-mismatch sample shows clean `ours=Approved` ↔ `raw=Denied` flips at `expected_fraud_score = 0.6` (the threshold), exactly where rank perturbation between methods has the most leverage. Refactor confirmed worth doing.
  - RAM cost of the planned refactor is **zero**: u16 quantization stays (the 84 MB win was from quantization, not normalization). Per-centroid `½‖c‖²` adds ~4 KB at load time. Per-row `½‖r‖²` is computed on the fly inside the existing SIMD loop (the 14 dequantized features are already in registers for `q·r`; just accumulate `r²` in the same pass — +14 fmas per row, no extra loads, fits within the existing 10× p99 headroom). Bumping `dataset_blob.VERSION` 3 → 4 is hygiene to invalidate stale v3 blobs (same bytes, different semantics — silent misreads otherwise). k-means switches from spherical (re-L2-normalize centroids each iter) to plain Euclidean (drop the renormalize, switch `assign_all` from max-dot to min-dist).
